# AeroLease Wise — UI Corporate Audit
**Audit Date:** 2026-03-11
**Auditor:** Claude Code (automated engineering review)
**Scope:** All 30 page components across three portals, shared layout components (Sidebar, Header, App.tsx)
**Portals Audited:** Admin (`/`), Flight School (`/portal/*`), Asset Owner (`/owner/*`)

---

## Executive Summary

The AeroLease Wise platform suffers from a **fundamental design system fragmentation** that undermines its corporate credibility. Three completely separate color palettes are used simultaneously across the application — each portal was built with a different visual identity. Beyond the color conflict, the audit found 9 dead/broken UI elements that either do nothing or produce incorrect output, multiple security-class issues including hardcoded developer PII in production code, and significant quality-tier disparities where different admin pages appear to have been built months apart.

The application is not production-ready from a UI/UX perspective in its current state. The issues below are ranked by severity and user-impact.

**Summary Scores by Portal:**

| Portal | Design Cohesion | Functional Completeness | Corporate Finish | Overall |

## Critical Issues

These issues cause direct functional failures, expose internal data, or violate security/professional standards. Fix before any production deployment.

### CRIT-01 — Hardcoded Developer Email in Production Guard Logic
**File:** `client/src/pages/admin-users.tsx` line 483
**Issue:** A specific personal email address (`zacharypurvis2@gmail.com`) is hardcoded in role-protection logic that prevents certain actions on user accounts. This is a PII leak and a brittle security dependency. If this email is ever changed or removed from the database, the guard silently breaks. Internal developer identity is exposed to anyone who reads source code (e.g., via sourcemaps).
**Severity:** Critical — Security / PII
**Fix:** Replace with a role-based check (`user.role === 'super_admin'`) and remove the hardcoded email entirely.

---

### CRIT-02 — Invoice PDF Contains Wrong Product Name
**File:** `client/src/pages/portal/payments.tsx` lines 96, 101, 224
**Issue:** PDFs generated by the "Download Invoice" feature print `"Aviation APE"` as the company name in the invoice header. The correct product name is "AeroLease Wise". Every invoice delivered to a flight school lessee is branded with a nonsensical placeholder name. This is a direct customer-facing defect.
**Severity:** Critical — Incorrect output delivered to end users
**Fix:** Replace all three occurrences of `"Aviation APE"` with `"AeroLease Wise"`.

---

### CRIT-03 — "Simulated Payment" Disclosed to Users in Toast
**File:** `client/src/pages/portal/payments.tsx` (payMutation onSuccess toast)
**Issue:** After the user submits a payment, the success toast message explicitly states "simulated payment" — revealing to users that the payment system is not real. This directly undermines trust in a financial platform.
**Severity:** Critical — Trust / Functional honesty
**Fix:** Either implement a real payment flow or remove the word "simulated" from the user-facing message entirely.

---

### CRIT-04 — Raw Markdown Rendered as Literal Text in Registration Success State
**File:** `client/src/pages/register.tsx` line 86
**Issue:** The registration success screen renders `**AeroLease Wise**` — the asterisks appear as literal characters on screen because JSX does not process markdown. Looks like a broken template.
**Severity:** Critical — Visible rendering bug on the registration confirmation screen
**Fix:** Replace with `<strong>AeroLease Wise</strong>` or use a `<b>` tag.

## High Priority Issues

These issues cause meaningful degradation to the user experience or create functional dead ends.

### HIGH-01 — Three Competing Brand Color Systems
**Files:** Throughout all pages
**Issue:** The application uses three completely separate color identities with no shared token:
- **Admin portal:** `brand` CSS variable (blue, defined in Tailwind config)
- **Auth pages (login/register):** `#6366f1` (indigo, hardcoded)
- **Sidebar, Portal and Owner portals:** `#007AFF` (iOS blue, hardcoded)

This means the same sidebar component that renders for all three user roles uses `#007AFF`, while the admin pages it wraps use the `brand` variable, and auth pages use `#6366f1`. A user moving between admin and portal views sees a completely different visual identity. The platform appears to have been assembled from three different design handoffs.
**Severity:** High — Core brand consistency
**Fix:** Define a single `--color-brand` CSS custom property. Standardize all three portals and auth pages to this token. Audit every hardcoded hex and replace.

---

### HIGH-02 — "Watch Experience" Button Is Dead (Landing Page)
**File:** `client/src/pages/landing.tsx` lines 65-72
**Issue:** The "Watch Experience" button renders prominently in the hero section with a `<Play>` icon but has no `onClick` handler. Clicking it does absolutely nothing. On a marketing landing page, this is the second primary CTA button — a dead CTA is a conversion killer.
**Severity:** High — Dead UI on the most-trafficked public page
**Fix:** Either attach a handler (modal video, YouTube embed, etc.) or remove the button until the feature exists.

---

### HIGH-03 — Nested Sidebar Layout Conflict in Settings
**File:** `client/src/pages/settings.tsx` line 208
**Issue:** The Settings page renders its own internal sidebar (`w-64 bg-gray-50 border-r sticky top-0 h-screen`) within a page that is already rendered inside the application's main sidebar layout. This produces a nested two-sidebar layout on desktop. The inner sidebar has `h-screen` which causes layout overflow. On mobile the inner sidebar stack is unworkable.
**Severity:** High — Layout-breaking on desktop and mobile
**Fix:** Replace the internal sidebar with a horizontal tab navigation component (shadcn `<Tabs>`), consistent with how most admin SaaS platforms handle settings.

---

### HIGH-04 — Admin Users Table Uses Raw HTML Instead of shadcn Table
**File:** `client/src/pages/admin-users.tsx`
**Issue:** This page builds its user table using raw `<table>`, `<thead>`, `<tr>`, `<td>` HTML elements, completely bypassing the `shadcn/ui` `<Table>` component used by every other page in the application. The visual output is styled inconsistently — column spacing, typography, and borders differ from all other tables.
**Severity:** High — Design consistency, component discipline
**Fix:** Refactor to use `<Table>`, `<TableHeader>`, `<TableBody>`, `<TableRow>`, `<TableHead>`, `<TableCell>` from `@/components/ui/table`.

---

### HIGH-05 — Admin Users Table Exposes Raw Database IDs
**File:** `client/src/pages/admin-users.tsx` line 429
**Issue:** The user table displays `ID: {user.id}` in the first column, directly exposing sequential internal database IDs to admin users. Sequential IDs are an enumeration attack vector and provide no UX value to administrators.
**Severity:** High — Security hygiene / UX
**Fix:** Remove the raw ID display. If a unique identifier is needed for admin reference, use a masked or shortened form (e.g., `#${String(user.id).padStart(5, '0')}`), or simply omit it.

---

### HIGH-06 — Portal Aircraft Detail Uses Wrong API Endpoint Key
**File:** `client/src/pages/portal/aircraft-detail.tsx` line 17
**Issue:** The portal aircraft detail page fetches from `/api/portal/browse/${id}` but the portal aircraft listing page (`my-aircraft.tsx`) fetches from `/api/portal/aircraft`. The endpoint inconsistency (`browse` vs `aircraft`) suggests this endpoint was intended for a public browsing flow that may no longer exist, or the detail page is hitting the wrong backend route entirely.
**Severity:** High — Potential data fetching failure / architectural inconsistency
**Fix:** Audit the backend to confirm which endpoint should serve the portal detail view and standardize to `/api/portal/aircraft/${id}`.

---

### HIGH-07 — Portal Maintenance Uses Different Aircraft Endpoint Than My Aircraft Page
**File:** `client/src/pages/portal/maintenance.tsx`
**Issue:** The maintenance page fetches its aircraft list from `/api/portal/my-aircraft` while the My Aircraft page itself fetches from `/api/portal/aircraft`. These are different endpoint names for what should be the same data. One of them may be hitting a non-existent or incorrectly scoped route.
**Severity:** High — API consistency / potential data mismatch
**Fix:** Standardize to one endpoint name across all portal pages. `/api/portal/aircraft` is preferred.

---

### HIGH-08 — App-Level Loading Spinner Uses No-Brand Raw CSS
**File:** `client/src/App.tsx` lines 60-66
**Issue:** The full-screen loading state shown to all users on every page load renders a raw `border-b-2 border-blue-600` spinner on a `bg-gray-50` background. This is completely unbrandned. It uses the generic Tailwind blue instead of the brand color, and `bg-gray-50` instead of white. As the first thing every user sees on every session, this should reflect the brand.
**Severity:** High — First impression / brand
**Fix:** Use the `brand` CSS variable for the spinner color, white background, and the AeroLease Wise wordmark.

## Medium Priority Issues

These issues degrade corporate finish and professionalism but do not cause complete functional failures.

### MED-01 — Landing Page Has Multiple Dead Navigation Links
**File:** `client/src/pages/landing.tsx` lines 22-26, 209-211
**Issue:** All six navigation links (Platform, Solutions, Enterprise in the nav; Privacy, Terms, Security in the footer) use `href="#"` anchor links that scroll to the top of the page. For a public-facing marketing page, dead navigation destroys credibility with prospective enterprise clients.
**Severity:** Medium
**Fix:** Implement destination sections with proper anchor IDs, or remove these links until the content exists.

---

### MED-02 — Stats Inconsistent Between Landing and Login Pages
**File:** `client/src/pages/landing.tsx` line 77 vs `client/src/pages/login.tsx` line 117
**Issue:** The landing page shows `$84.2b` in assets managed; the login page shows `$4.2b`. Both are displayed prominently as platform statistics. The 20x discrepancy makes the platform look careless.
**Severity:** Medium
**Fix:** Align all statistics to a single agreed value across all pages.

---

### MED-03 — Copyright Year Is Two Years Stale
**File:** `client/src/pages/landing.tsx` line 208
**Issue:** Footer reads "© 2024 AeroLease Wise." Current year is 2026. A stale copyright year signals a neglected product.
**Severity:** Medium
**Fix:** Use `new Date().getFullYear()` dynamically.

---

### MED-04 — Owners and Lessees Pages Are a Visual Downgrade from Aircraft Page
**Files:** `client/src/pages/owners.tsx`, `client/src/pages/lessees.tsx` vs `client/src/pages/aircraft.tsx`
**Issue:** The aircraft page uses refined semantic color tokens (`text-[#1e293b]`, `text-[#64748b]`), `rounded-2xl` search containers, Skeleton loading states, and `max-w` containers. The owners and lessees pages use raw Tailwind color utilities (`text-gray-900`, `text-gray-500`), raw `animate-pulse` divs for loading (not the `<Skeleton>` component), no `max-width` wrapper, and inconsistent border-radius on cards. These pages appear to have been built by a different developer or at a different quality standard.
**Severity:** Medium
**Fix:** Bring owners.tsx and lessees.tsx up to the design standard of aircraft.tsx: use semantic tokens, `<Skeleton>` for loading, consistent container width.

---

### MED-05 — Payments Page Has 11-Column Table with Free-Text Period Field
**File:** `client/src/pages/payments.tsx`
**Issue:** The payments table has 11 columns: Reference, Aircraft, Lessee, Period, Type, Gross, Commission, Net, Status, Date, Action. This will overflow horizontally on most laptop viewports. Additionally, the Period field is a free-text input with placeholder `"e.g. January 2024"` — meaning data entry is completely non-standardized. Some rows legitimately show `—` in Gross/Commission/Net columns with no explanation or tooltip, which looks like broken display to users.
**Severity:** Medium
**Fix:** Reduce visible columns to 6-7 (defer less critical columns to a row-expand detail view). Replace the free-text period field with a `<Select>` or month picker component. Add a tooltip or legend explaining the `—` display.

---

### MED-06 — Password Reset Is a Dead End
**File:** `client/src/pages/login.tsx` line 172-176
**Issue:** The "Reset Link" button on the login page only shows a toast saying "Please contact your administrator to reset your password." There is no actual reset flow, no email link, and no administrator contact mechanism. For a self-registration platform, this is a functional gap that will result in locked-out users.
**Severity:** Medium
**Fix:** Implement a forgot-password email flow, or if intentionally admin-managed, provide a visible email address or contact form rather than a dead button.

---

### MED-07 — Portal Payments "Download Full Policy" Is Permanently Disabled
**File:** `client/src/pages/portal/payments.tsx`
**Issue:** The "Download Full Policy" button is rendered with `disabled` and `cursor-not-allowed`. There is no tooltip, no explanation, and no indication of when this feature will be available. A permanently disabled button with no explanation is confusing to users.
**Severity:** Medium
**Fix:** Either implement the feature or hide the button entirely if it is not ready. If it's planned, add a tooltip ("Coming soon") and a non-disabled visual state that indicates its future availability.

### MED-08 — Help & Support Page Contains Fake Placeholder Contact Information

**Issue:** The page displays `1-800-AERO-LEASE`, `123 Aviation Way, Miami FL 33142` as real contact information visible to all users. These are clearly placeholder values that were never replaced with actual company contact details.

### MED-09 — Permanent "Email Notifications Being Configured" Banner
**File:** `client/src/pages/help-support.tsx`
**Issue:** An informational banner with the text "Email notifications are currently being configured" appears permanently on every load for all users. This appears to be a temporary development note that was never removed.
**Severity:** Medium

### MED-10 — Wrong Icon for "External URL" Button in Admin Documents
**File:** `client/src/pages/documents.tsx` line 683
**Issue:** The button to open a document's external URL uses the `<Grid3X3>` icon (a 3×3 grid layout icon). This is semantically incorrect — the expected icon for an external link is `<ExternalLink>` from Lucide.

### MED-11 — Maintenance Page "Reported By" Shows "Admin" as Literal String
**File:** `client/src/pages/maintenance.tsx` line 443
**Issue:** System-created maintenance records display the literal string `"Admin"` in the "Reported By" column rather than the actual user who created the record. This provides no audit trail value.
**Severity:** Medium
**Fix:** Resolve the actual user name from the `reportedBy` field's user ID reference, or store the user's full name at record creation time.

### MED-12 — accept-invite.tsx Uses Third Brand Color (#007AFF vs auth flow #6366f1)
**File:** `client/src/pages/accept-invite.tsx`
**Issue:** The accept-invite page uses `#007AFF` (iOS blue) while the login and register pages that bracket it in the auth flow use `#6366f1` (indigo). A user arriving via invite email sees a different brand color than they saw on the login page they're directed back to.
**Severity:** Medium
**Fix:** Standardize accept-invite to use `#6366f1` (same as login/register) until the broader three-palette issue is resolved.

---

### MED-13 — Portal Documents and Admin Documents Reference Different Date Fields
**File:** `client/src/pages/portal/documents.tsx` vs `client/src/pages/documents.tsx`
**Issue:** Admin documents displays `formatDate(doc.uploadDate)`, while portal documents displays `formatDate(doc.createdAt)`. These are different fields on the same document object. One of these is reading a field that may not exist, resulting in `undefined` being passed to `formatDate()`.
**Severity:** Medium
**Fix:** Audit the document schema — pick one canonical date field and use it consistently across all document views.

---

### MED-14 — "Scheduling" Quick Action Links to Hour Logging (Misleading Label)
**File:** `client/src/pages/portal/my-aircraft.tsx` lines 116-125
**Issue:** A quick action card labeled "Scheduling" navigates to `/portal/hour-logging`. Hour logging and scheduling are conceptually different features. Users expecting a scheduling view (calendar, availability management) are taken to a flight-hours entry form instead.
**Severity:** Medium
**Fix:** Rename the quick action to "Log Hours" or "Flight Hours", or implement a separate scheduling view.

---

### MED-15 — `formatMonth` Utility Function Duplicated Across Multiple Files
**Files:** `client/src/pages/owner/revenue.tsx` line 20, `client/src/pages/owner/aircraft-detail.tsx` line 4
**Issue:** The `formatMonth(ym: string): string` function is defined identically in two separate files. Any bug or locale change requires updates in multiple places.
**Severity:** Medium — Code quality / maintainability
**Fix:** Move `formatMonth` to `client/src/lib/utils.ts` and import from there.
